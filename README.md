# FinRAG

A fully offline Retrieval-Augmented Generation (RAG) chatbot for PDF documents. Built as an internship project at FinSeal, the long-term goal is a private assistant for financial filings (annual reports, prospectuses, regulatory disclosures) that runs entirely on local infrastructure — no API keys, no data egress.

The current codebase is **Phase 1**: plain-text PDFs (project reports), used to validate the retrieval + generation pipeline. **Phase 2** extends ingestion to scanned pages (OCR) and structured tables (table extraction), targeting real financial documents.

---

## Stack

| Layer            | Choice                                                       |
| ---------------- | ------------------------------------------------------------ |
| LLM              | `llama3.1:8b` served via [Ollama](https://ollama.com)        |
| Embeddings       | `nomic-embed-text` (768-dim) via Ollama                      |
| Vector store     | [ChromaDB](https://www.trychroma.com), persisted to disk     |
| PDF parsing      | [PyMuPDF](https://pymupdf.readthedocs.io) (`fitz`)           |
| Chunking & chain | [LangChain](https://python.langchain.com) (core + community) |
| UI               | [Streamlit](https://streamlit.io)                            |

Everything runs locally; the only network calls are localhost requests to the Ollama daemon at `http://localhost:11434`.

## Configuration (`config.py`)

| Setting           | Default                | Notes                                                                |
| ----------------- | ---------------------- | -------------------------------------------------------------------- |
| `LLM_MODEL`       | `llama3.1:8b`          | Any Ollama-compatible chat model                                     |
| `EMBEDDING_MODEL` | `nomic-embed-text`     | Must match between ingest and query                                  |
| `OLLAMA_BASE_URL` | `http://localhost:11434` |                                                                    |
| `CHUNK_SIZE`      | `1000`                 | Characters per chunk                                                 |
| `CHUNK_OVERLAP`   | `200`                  | Overlap to preserve context across chunk boundaries                  |
| `COLLECTION_NAME` | `finrag`               | Chroma collection identifier                                         |
| `TOP_K`           | `8`                    | Chunks returned to the LLM                                           |
| `MMR_FETCH_K`     | `24`                   | Candidate pool MMR diversifies over                                  |
| `MMR_LAMBDA`      | `0.7`                  | 1.0 = pure relevance, 0.0 = pure diversity                           |

Changing `EMBEDDING_MODEL` or `CHUNK_SIZE` requires deleting `vectorstore/` and re-running `ingest.py`.

## Ingestion pipeline (`ingest.py`)

1. **Discover** all `data/*.pdf`.
2. **Extract author from filename** — `author_from_filename()` strips B-code prefixes (`B076_`) and report suffixes (`_FPR`, `_FinalReport`), splits the remainder on underscores and camelCase, yielding e.g. `B076_RiyanshSachdev_FPR.pdf` → `Riyansh Sachdev`.
3. **Load pages** with PyMuPDF. Each non-empty page becomes a LangChain `Document` whose `page_content` is prefixed with a header:
   ```
   Document: <filename>
   Author: <author>

   <page text>
   ```
   This embeds the author name into every chunk, allowing semantic queries like "Vikhyat's methodology" to match Vikhyat's pages even when his name doesn't appear in the body.
4. **Chunk** with `RecursiveCharacterTextSplitter` (1000/200, separator order `["\n\n", "\n", ". ", " ", ""]`).
5. **Embed** each chunk with `OllamaEmbeddings` and **persist** to `vectorstore/` under the `finrag` collection. Metadata retained per chunk: `source` (filename), `author`, `page`.

Run:

```bash
source .venv/bin/activate
python ingest.py
```

Re-running rebuilds against whatever is currently in `data/`. Wipe `vectorstore/` first if you want a clean slate.

## Query pipeline (`query.py`)

```
question
   │
   ├── detect_source_filter(question)
   │        ↳ if exactly one known author name appears → return filename
   │
   ├── get_retriever(source_filter)
   │        ↳ Chroma MMR (k=8, fetch_k=24, λ=0.7), filtered by `source` if provided
   │
   ├── retriever.invoke(question) → List[Document]
   │
   ├── format_context(docs)
   │        ↳ "[i] (filename p.N)\n<text>"  for each chunk, joined with blank lines
   │
   └── PROMPT | ChatOllama(llama3.1) | StrOutputParser
            ↳ returns answer grounded in context, with explicit "I don't know" fallback
```

`detect_source_filter` keeps a name → filename index built from `data/`. A question is filtered only if its tokens match **exactly one** known author (avoids ambiguity from common terms or multi-author queries).

The MMR retriever uses Maximal Marginal Relevance to balance top-similarity chunks with diversity, preventing N near-duplicate chunks from a single boilerplate page (e.g. completion certificates) from monopolizing the context window.

CLI:

```bash
python query.py "summarize Riyansh's project"
```

Programmatic:

```python
from query import ask
result = ask("what tech stack did Prakarsh use")
# result = {"answer": str, "sources": List[Document], "filtered_to": str | None}
```

## Streamlit UI (`app.py`)

Chat interface at `http://localhost:8501`. Per turn:

1. Detect source filter from the question; cache a retriever per filter value via `@st.cache_resource`.
2. Run retrieval → format context → invoke the chain.
3. Render: optional "Filtered to: …" caption, the answer (markdown), and an expandable **Sources** panel listing each chunk's `filename — p.N` and the raw text.

Session history is kept in `st.session_state.history` for the duration of the tab.

Run:

```bash
source .venv/bin/activate
streamlit run app.py
```

First-run prompts are suppressed via `~/.streamlit/config.toml` (`headless = true`, `gatherUsageStats = false`).

## Design decisions worth flagging

- **Author header injection at ingest time** is intentional. Without it, embeddings for "Vikhyat's project" matched generic project-language chunks across all PDFs rather than Vikhyat's own report.
- **Hard source filtering on author detection** is more reliable than embedding-only retrieval for name-scoped questions. It complements rather than replaces semantic search.
- **MMR over pure similarity** prevents low-value boilerplate (cover pages, completion certificates, identical disclaimers) from saturating the top-K.
- **Strict grounding in the prompt** — the LLM is told to say "I don't know" rather than hallucinate. This is the correct default for financial documents in Phase 2, where invented numbers are unacceptable.
- **No re-ranking, no query rewriting, no agent loop** — this is a deliberately small RAG. Complexity is added only when a measurable failure justifies it.

## Known limitations (Phase 1)

- **Scanned PDFs are invisible.** PyMuPDF's `get_text()` returns nothing for image-only pages, so any such PDF yields 0 chunks. Example in the current corpus: `B016_JashUpadhyay_FPR.pdf`. Phase 2 addresses this.
- **Tables are flattened to whitespace-separated text.** Multi-column financial tables lose their row/column structure when extracted by PyMuPDF. Phase 2 addresses this.
- **No conversational memory** — each question is answered from retrieval alone; follow-ups like "and what about the second one" won't resolve coreferences.
- **No citations inside the answer text** — sources are shown in a separate panel rather than inline `[1]` markers.
- **Single-collection index.** All PDFs live in one Chroma collection; per-document or per-domain collections would help if the corpus grows past a few hundred files.

## Phase 2 — Financial documents

Goal: ingest real corporate financial filings (annual reports, 10-Ks, prospectuses) and answer questions over their narrative *and* their tabular data, including pages that are scanned images.

### 1. OCR for scanned / image-only pages

- **Engine:** [Tesseract OCR](https://github.com/tesseract-ocr/tesseract) (open source, offline). Install via `brew install tesseract` (macOS) or `apt-get install tesseract-ocr` (Linux).
- **Python binding:** `pytesseract` (subprocess wrapper) or PyMuPDF's built-in `page.get_textpage_ocr()` which calls Tesseract via the C API.
- **Strategy:** In `load_pdf()`, attempt `page.get_text()` first. If the result is empty *and* the page contains raster images (`page.get_images()` is non-empty), render the page to a pixmap at 300 DPI and run it through Tesseract. Tag OCR-derived chunks in metadata (`"ocr": True`) so we can later evaluate quality separately.
- **Caveats:** OCR is lossy on dense tables and small fonts; expect to tune DPI and consider page-level pre-processing (deskew, binarise) with `opencv-python` for low-quality scans.

### 2. Table extraction

Plain-text extraction destroys table semantics. Two complementary libraries:

- **[pdfplumber](https://github.com/jsvine/pdfplumber)** — pure Python, good at detecting table grids from vector-drawn lines. Best for born-digital PDFs where the table has visible borders.
- **[Camelot](https://camelot-py.readthedocs.io)** — two flavours: `lattice` (uses ruling lines, similar to pdfplumber) and `stream` (uses whitespace, better for borderless tables). Requires Ghostscript.
- **Fallback:** for scanned tables, OCR the page with Tesseract in `--psm 6` (assume uniform block of text) and use heuristic column detection, or use a layout model (e.g. `img2table`, LayoutParser) if accuracy matters.

**Ingestion strategy:**
1. For each page, attempt table extraction first.
2. If tables are found, serialise each as Markdown (`| col | col |` rows) and store as its own `Document` with metadata `{"type": "table", "source": ..., "page": ...}`. Markdown tables are tokenised cleanly by the embedder and render nicely if the LLM quotes them back.
3. Extract the page's non-tabular narrative text separately, store as `{"type": "text", ...}`.
4. Chunk text normally; tables are stored whole (do not split a table mid-row).

### 3. Retrieval changes

- **Hybrid metadata filtering:** add `type` to Chroma queries so questions about numbers ("what was revenue in FY24") can be biased toward `type: "table"` chunks.
- **Re-ranking:** consider a cross-encoder re-ranker (e.g. `bge-reranker-base` via `sentence-transformers`) on the top-`fetch_k` to push numerical/tabular evidence above generic narrative when the question is quantitative.
- **Citation in answer:** switch the prompt to require inline `[source p.N]` markers so quoted numbers are traceable to a specific page.

### 4. Evaluation

Phase 2 needs a small evaluation harness — a YAML of `(question, expected_answer_substring, expected_source_file)` triples, executed against `ask()`, scored on retrieval hit rate and answer-substring match. This catches regressions when changing chunking, prompts, or retrieval parameters.

## Troubleshooting

| Symptom                                              | Fix                                                                                                  |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `ConnectionError` from `OllamaEmbeddings`            | Ensure `ollama serve` is running and both models are pulled.                                         |
| `Collection 'finrag' has 0 vectors`                  | `data/` is empty or all PDFs are scanned. Add text PDFs or wait for Phase 2 OCR.                     |
| Streamlit prompts for email on first run             | A `~/.streamlit/config.toml` with `headless = true` suppresses it.                                   |
| Answers reference a PDF you've since removed         | Wipe `vectorstore/` and re-run `ingest.py`.                                                          |
| Retrieval returns boilerplate (cover, certificates)  | Increase `MMR_LAMBDA` toward 1.0 (more relevance) or lower it toward 0.0 (more diversity).           |
| "I don't know" on a question you know is answerable  | Check the Sources panel — if the right page isn't retrieved, the embedding isn't matching; rephrase or widen `TOP_K`. |
